# Phase 07: PR

## Objective

Prepare and submit a pull request that passes review, with complete context for the reviewer.

## Input

- Approved QA report (`qa-report/YYYY-MM-DD-[slug].md`)
- All code changes ready in branch

## Output

- Pull request created and ready for review
- PR description with complete context

## PR Preparation Steps

### 1. Verify Branch is Clean

```bash
git status        # No uncommitted changes
git log --oneline # Only relevant commits
```

### 2. Write PR Description

```markdown
## Summary
[Brief description of what this PR does]

## Changes
### Added
- [ ] Feature: [Description]

### Modified
- [ ] [File]: [What changed]

### Removed
- [ ] [File/Feature]: [What was removed]

## Related Issues
- Closes #[issue-number]

## Testing
- [ ] Unit tests: [X] passing
- [ ] Integration tests: [X] passing
- [ ] E2E tests: [X] passing

## Checklist
- [ ] Code follows project conventions
- [ ] Documentation updated
- [ ] No breaking changes (or documented)
- [ ] PR size: [XS/S/M/L/XL] (aim for M or smaller)
```

### 3. Self-Review Checklist

Before opening PR, verify:

- [ ] PR description is clear and complete
- [ ] All tests passing locally
- [ ] No debug code or TODO comments left
- [ ] PR size is manageable (split if XL)
- [ ] Related documentation updated

## PR Size Guidelines

| Size | Lines Changed | Action |
|---|---|---|
| XS | < 50 | Perfect, instant review |
| S | 50-200 | Normal review |
| M | 200-500 | Review carefully |
| L | 500-1000 | Consider splitting |
| XL | > 1000 | Must split |

**If PR > 500 lines**: Split into multiple PRs by module or feature.

## Agent Behavior in This Phase

**Role**: Developer (PR Author)

**Tasks**:
1. Verify branch is clean and rebased
2. Write comprehensive PR description
3. Run final checks (lint, test, build)
4. Split PR if too large
5. Address reviewer comments
6. Ensure PR is mergeable

**Do NOT**:
- Open PR with incomplete description
- Leave failing CI
- Ignore review comments
- Merge without approval
